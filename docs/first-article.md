## 正文

### 开头

2024年，我帮一家中型企业过等保三级评审。

评审老师翻到HR模块的时候问了一句话："员工的入职、调岗、离职记录，你们怎么保证没有被篡改？"

对方HR答："我们有OA审批流。"

评审老师笑了笑，没说话。

那一刻我意识到——在中国，绝大多数企业的HR管理，在**审计合规**这个维度上，是裸奔的。

HR数据太容易被篡改了。一个懂数据库的管理员，改一条薪资记录只需要3秒钟。而且**没有任何证据证明它被改过**。

然后我就想：能不能用区块链做一层**不可篡改的审计存证层**？

不是替代现有HR系统，而是在现有系统上加一层"证据链"。

于是有了 **D-HRS（链聘通）**。

---

### 问题

中国企业在HR合规审计上有4个核心痛点：

**1. 操作不可追溯**
- 调薪是谁批的？什么时候批的？
- 截图？截图能当证据吗？
- 数据库日志？DBA自己就能改

**2. 资质文件造假**
- 假学历、假履历，背调成本 ¥500/人
- 验证靠打电话/发邮件，3-5天

**3. 等保三级难通过**
- 等保三级要求"数据存储的完整性和不可否认性"
- 传统数据库做不到
- 请咨询公司做一套制度文档就要 ¥5-15万

**4. 法规多且互相冲突**
- 924通知：不能用加密货币
- PIPL：不能上链个人敏感信息
- 区块链信息服务管理规定：需要实名认证+备案
- 想做合规，光研究法规就要一个月

---

### 方案

我的方案是：不做HR系统，做**合规存证中间件**。

```
┌─────────────────────────────────────────────┐
│    企业现有系统（北森/Moka/钉钉/自研）        │
└──────────────────┬──────────────────────────┘
                   │ API 对接
┌──────────────────▼──────────────────────────┐
│        D-HRS 合规存证层                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ KYC实名  │  │ 操作上链  │  │ 制裁名单  │  │
│  └──────────┘  └──────────┘  └──────────┘  │
└──────────────────┬──────────────────────────┘
                   │ 哈希上链
┌──────────────────▼──────────────────────────┐
│     智能合约层（Solidity 0.8.19）              │
│  ComplianceEngine + EmployeeRegistry + ...   │
└─────────────────────────────────────────────┘
```

核心设计原则：

1. **不上链任何敏感数据**——只上链操作哈希，原始数据用AES-256加密存储在本地数据库
2. **不发币**——纯合规工具，没有ERC-20，没有加密货币支付
3. **实名准入**——所有操作必须经过KYC+制裁名单筛查
4. **标准兼容**——W3C VC标准发行可验证凭证

### 合规设计（这可能是我写的最多代码的地方）

整个系统最复杂的不是业务逻辑，而是合规。

```solidity
// ComplianceEngine.sol - 每个操作前检查
function guard(address account) external view {
    require(kycStatus[account] == KYCStatus.VERIFIED, "KYC required");
    require(!sanctions[account], "Address sanctioned");
}
```

任何一个写入操作之前，合约层会检查：
1. 这个地址是否完成了实名认证？
2. 这个地址是否在制裁名单里？
3. 当前操作是否在限频范围内？

后端同时同步检查：
1. 是否有有效的JWT Token？
2. 内容是否通过审核过滤？
3. 操作是否记录到审计日志？

**双层守卫，无一遗漏。**
